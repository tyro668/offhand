# 跨应用语音输入的上下文记忆设计

## 背景

Offhand 现在已经开始具备“术语纠正记忆”能力：

- 用户修改历史听写结果
- 系统抽取术语候选
- 候选进入待确认池
- 接受后写入词典

这解决的是“词”和“术语”的问题，但还没有解决另一类很高频的问题：

- 用户说到一半时，系统不知道他最近在写什么主题
- 相同短语在不同上下文里应有不同表达
- 同一句口语，在不同应用和任务里应该补成不同风格
- 同样的代词、简称、缩写，只有结合最近历史才知道具体指什么

例如：

- 用户刚连续输入了三条和“招聘 JD”相关的内容，再说一句“这个再强调一下要有大模型经验”
- 用户刚在飞书里写项目同步，再说一句“这个先不发，等明天再同步”
- 用户刚在邮件里写英文商务回复，再说一句“语气再柔和一点”

这些都不是“术语错了”，而是“缺上下文”。

所以，除了术语修正，我们还需要一层新的能力：

- 利用已有历史记录，为当前听写提供主题、风格、指代和续写方向上的上下文优化。

---

## 要解决的问题

### 当前系统的不足

当前链路里，历史记录主要用于：

- 回看
- 手工编辑
- 沉淀术语

但历史记录还没有真正参与下一次听写的生成过程。

这会带来几个问题：

1. 主题不连续
   - 用户刚输入一段产品需求，下一句仍在讲同一个需求，但系统把它当成全新上下文处理。

2. 风格不连续
   - 用户前面都在写正式邮件，后面一句还是同一封邮件里的补充，系统却按普通口语处理。

3. 指代不清
   - “这个”“那个版本”“上面那条”“那个接口”都需要从最近记录里找参照。

4. 句子完成度不足
   - 用户口述天然省略成分，只有结合刚刚几条历史，才能补出更完整、更自然的书面表达。

---

## 目标

### 功能目标

- 基于历史听写记录，构建轻量级“上下文记忆”。
- 在每次新听写时，自动检索最相关的近期上下文。
- 将上下文作为增强信号传给纠错/增强链路，提升表达完整度与一致性。
- 让结果更像“延续当前写作任务”，而不是“孤立的一句转写”。

### 体验目标

- 用户无需主动选择“我现在在写什么”。
- 系统默认聪明，但不能过度自作主张。
- 优化要以“更连续、更自然”为主，而不是强行改写用户原意。

### 非目标

- 本期不做长文档级的全文理解。
- 不做跨应用读取用户当前编辑器内容。
- 不做云端知识库或企业 RAG。
- 不把历史记录直接发散成通用聊天记忆。

---

## 核心概念

### 1. Context Memory

从历史记录中提炼出的上下文片段，用于帮助下一次听写理解：

- 当前主题
- 当前写作对象
- 当前表达风格
- 最近明确提到的实体/对象

### 2. Context Recall

在一次新听写开始或结束时，从最近历史中挑出最相关的若干条记录，组成“上下文参考集”。

### 3. Context Hints

不是把整段历史直接丢给模型，而是先压缩成结构化提示：

- 最近主题：招聘、日报、Bug 同步、客户邮件
- 表达风格：正式、简洁、口语、英文商务
- 最近实体：候选人、版本 3.2、OpenAPI 网关、深圳办公室

## 典型用户故事

### 用户故事 1：同一主题续写

历史里刚刚有两条：

- “这个岗位需要负责 Agent 平台的工程化建设”
- “最好有 LLM 应用落地经验和工作流设计经验”

用户接着说：

- “然后再补一句要有跨团队协作能力”

系统应倾向输出：

- “然后再补充一句，需要具备较强的跨团队协作能力。”

而不是把它当成一条完全独立的口语句子。

### 用户故事 2：同一封邮件中的风格延续

前面几条都很正式：

- “感谢你的及时反馈”
- “我们会在本周内完成内部确认”

用户继续说：

- “这个先别发得太强硬”

系统更应该理解为：

- “这部分措辞先不要太强硬。”

### 用户故事 3：代词消解

前面刚说过：

- “OpenAI 的新接口我们先只接 Chat Completions”

接着说：

- “这个先不要全量放开”

系统如果能带着上下文，就更容易理解“这个”指的是前面提到的接口策略，而不是任意泛指。

## 设计原则

1. 历史记录只做“辅助理解”，不替代原始语音内容
2. 近因优先，最近几条记录比很久之前的记录更重要
3. 相似任务优先，只召回与当前内容高度相关的上下文
4. 可控可退化，没有高置信上下文时就按当前单句处理
5. 与术语记忆分层，术语解决“字词正确”，上下文解决“表达更对”

---

## 能力边界

这套能力适合解决：

- 句子续写与补全
- 风格延续
- 代词和简称的局部消解
- 同一任务中的上下文一致性

这套能力不适合解决：

- 用户完全换话题后的强行延续
- 数天前旧上下文的硬召回
- 用户隐私敏感内容的过度推断
- “替用户做决定”式改写

---

## 总体方案

在现有链路上增加一层“上下文召回与压缩”：

1. 历史记录继续按 `Transcription` 存储
2. 每条历史在落库后异步生成一个轻量 `ContextProfile`
3. 新听写结束后，在增强阶段根据当前文本检索最近相关历史
4. 将相关历史压缩成 `ContextHints`
5. 把 `ContextHints` 注入现有增强/纠错 prompt
6. 最终输出更符合最近任务上下文的文本

---

## 数据分层

### 原始层：History

直接复用现有 `Transcription`

- `id`
- `text`
- `rawText`
- `createdAt`
- `provider`
- `model`

### 特征层：ContextProfile

为每条历史构建一个轻量画像：

- `historyId`
- `summary`
  - 这条记录的一句话摘要
- `topicTags`
  - 例如 `招聘`、`日报`、`邮件`、`技术方案`
- `styleTag`
  - `formal` / `casual` / `email` / `english`
- `entities`
  - 从文本里提取的关键对象，如产品名、人名、版本号
- `embedding`
  - 可选，MVP 可先不做向量，只做规则召回
- `createdAt`

### 运行层：ContextHints

每次听写时动态生成：

- `recentTopic`
- `recentStyle`
- `relatedEntities`
- `referenceTexts`
  - 最多 2-3 条

## MVP 检索策略

第一期不需要直接上向量检索，可以做“轻量规则召回”。

### 召回信号

1. 时间接近
   - 最近 10 条或最近 30 分钟的历史优先

2. 词面重叠
   - 当前文本与历史文本共享关键词

3. 术语命中
   - 当前文本命中了某些词典词或候选词

4. 风格继承
   - 最近几条都呈现相近风格时，作为默认风格

### 排序公式

可用一个简单打分：

- `score = recencyScore + overlapScore + entityScore + styleScore`

只取前 2-3 条，避免提示词过长。

---

## ContextProfile 生成策略

可以分两期做：

### 第一期：规则提取

从文本里直接提取：

- 是否像邮件
- 是否像英文
- 是否像任务描述
- 是否包含明显实体（版本号、英文名、产品名）

优点：

- 快
- 成本低
- 可本地完成

### 第二期：LLM 压缩

在空闲时异步调用一个小模型，把历史记录压缩成：

- 主题
- 风格
- 关键对象
- 是否适合作为后续上下文

优点：

- 质量更高
- 对口语句子理解更强

## 与现有链路的集成点

### 方案 A：接到 AI Enhance 前

链路：

- STT
- Correction
- Context Recall
- AI Enhance

适合：

- 主要优化最终表达

优点：

- 复用现有增强阶段
- 改动最小

### 方案 B：同时服务 Correction 和 AI Enhance

链路：

- STT
- Context Recall
- Correction
- AI Enhance

适合：

- 上下文也参与纠错判断

建议：

- MVP 先走方案 A
- 后续再逐步前移到 Correction 阶段

## Prompt 注入方式

不要直接把整段历史粗暴拼进去，而是注入结构化提示：

```text
【最近输入上下文】
- 最近主题：招聘 JD
- 最近表达风格：正式、用于岗位描述
- 最近提到的对象：Agent 平台、LLM 应用落地、跨团队协作
- 参考表达：
  1. 这个岗位需要负责 Agent 平台的工程化建设
  2. 最好有 LLM 应用落地经验和工作流设计经验

请在不改变用户原意的前提下，尽量保持与上述上下文一致的表达风格和主题延续性。
```

这样模型更容易“延续上下文”，而不是被长文本干扰。

## UI 设计

### 历史页

在历史记录卡片上增加一个轻量标识：

- `已用于上下文记忆`

或者在详情弹窗里展示：

- 主题标签
- 风格标签
- 关键对象

### 设置页

增加一个开关：

- `启用历史上下文优化`

可选再加两个滑杆/选项：

- 回看最近几条记录
- 仅在 AI 增强开启时生效

### 结果页或悬浮反馈

当一次结果明显用到了上下文时，可显示轻提示：

- `已参考最近输入上下文`

注意：

- 这是增强感知，不是必须功能
- 不要频繁打扰

## 隐私与安全

这是一个高敏感能力，必须克制。

### 风险

- 历史中可能包含密码、地址、电话、客户信息
- 把不相关旧内容带到新任务里，可能造成误改
- 用户可能不希望不同任务之间互相“串上下文”

### 约束

1. 默认只看近期记录
2. 只召回少量高相关内容
3. 敏感模式下可完全关闭
4. 后续可增加“应用隔离上下文”
   - 比如浏览器里的上下文不自动用于微信输入

class ContextProfile {

  final List<String> topicTags;
  final String? styleTag;

Provider / Service 层建议新增：

- `ContextMemoryService`
- `ContextRecallService`

职责拆分：

- `ContextMemoryService`
  - 为历史记录生成和维护 `ContextProfile`

- `ContextRecallService`
  - 输入当前文本
  - 输出 `ContextHints`

## 分阶段落地建议

### Phase 1：规则版上下文续写

- 为历史记录生成简单 `ContextProfile`
- 每次增强前从最近记录里挑 2-3 条高相关文本
- 将结构化上下文提示注入 AI Enhance prompt

这是最小可交付版本。

### Phase 2：上下文摘要与风格识别

- 增加主题标签、风格标签、关键对象提取
- 提升召回质量

### Phase 3：应用级隔离与个性化

- 针对不同应用维护不同上下文池
- 比如邮件、IM、文档、代码场景各自更稳定

---

## 与术语记忆的关系

两者不要混为一谈。

### 术语记忆

解决：

- “反软” 应该是 “帆软”
- “open api” 应该是 “OpenAPI”

关注点：

- 字词正确性
- 长期稳定映射

### 上下文记忆

解决：

- 这一句是在延续刚才的招聘 JD、日报还是邮件
- 这一句应该用什么语气、什么写法、指的是什么对象

关注点：

- 任务连续性
- 表达完整度
- 风格一致性

最好的方式是两层同时存在：

- 术语记忆保底“词对”
- 上下文记忆提升“句对”

---

## 建议的 MVP 范围

如果现在开始做，建议先只做下面这些：

1. 新增 `历史上下文优化` 开关
2. 从最近历史中召回 2-3 条高相关记录
3. 只在 AI Enhance 阶段注入 `ContextHints`
4. 不做向量，不做云端，不做应用隔离
5. 在 debug 日志里打印本次命中的上下文来源

这样风险最小，也最容易验证收益。

---

## 成功指标

可以用这些指标判断是否值得继续：

1. 用户对历史结果的再次改写率是否下降
2. 连续输入场景下的句子完整度是否提升
3. 同一任务中的风格一致性是否提升
4. 用户是否更少去手工补全代词、省略项

---

## 一句话总结

术语记忆是在教 Offhand “这个词应该怎么写”，  
上下文记忆是在教 Offhand “你现在大概率正在写什么，以及这句话应该怎么顺着写下去”。
